## Context

当前 VNote 的 Gitee 同步机制基于 `GiteeSyncService`，通过 `pushFile()` 方法将本地修改推送到 Gitee 仓库。存在以下问题：

1. **触发路径多且分散**：Ctrl+S、自动保存、窗口关闭、新建文件、配置更新都会触发同步
2. **无防抖机制**：同一文件短时间内多次修改会产生多个同步任务
3. **SHA 缓存问题**：pullFile 后 SHA 未更新，导致后续 pushFile 失败（409 Conflict）
4. **状态不透明**：用户无法了解同步任务执行情况
5. **并发冲突**：多个同步任务同时执行时可能产生竞争条件

## Goals / Non-Goals

**Goals:**
- 减少无效的 Gitee API 调用，降低触发失败率
- 提供清晰的用户反馈，展示同步状态和进度
- 支持批量合并和任务去重，优化持续修改场景
- 提供离线恢复能力，保证数据可靠性

**Non-Goals:**
- 不修改 Gitee API 本身
- 不提供完整的版本控制功能（保留 Gitee 原生的 commit 历史）
- 不支持跨文件的事务性同步（每个文件独立同步）

## Decisions

### 决策 1：智能同步调度器架构

**选择**：采用事件路由 + 优先级队列 + 防抖机制

```
事件路由：
- Ctrl+S ──▶ 高优先级 ──▶ 立即队列 (debounce=0)
- 自动保存 ──▶ 低优先级 ──▶ 批量队列 (debounce=batchWindowMs)
- 新建文件 ──▶ 最高优先级 ──▶ 立即队列
```

**理由**：
- Ctrl+S 是用户显式意图，应立即执行
- 自动保存频繁触发，需要批量合并减少 API 调用
- 事件路由使逻辑清晰，易于扩展

**替代方案**：
- 单一队列 + 定时批量处理：不够灵活，无法区分优先级
- 无队列直接推送：容易产生并发冲突

### 决策 2：文件状态机

**选择**：五状态模型

| 状态 | 含义 | 状态指示 |
|------|------|----------|
| SYNCED | 已同步 | 🟢 |
| PENDING | 待同步（防抖中） | 🟡 |
| SYNCING | 同步中 | 🔵 |
| FAILED | 失败（等待重试） | 🔴 |
| PAUSED | 暂停 | ⚪ |

**理由**：
- 状态机提供清晰的同步生命周期管理
- 用户可通过状态指示器快速了解同步情况
- 状态转换规则简单，易于实现和测试

### 决策 3：批量合并策略

**选择**：可配置时间窗口（默认 3000ms，范围 1000-10000ms）

**理由**：
- 用户场景：5 分钟保存 2-3 次，3 秒窗口可有效合并
- 可配置性：不同用户可能有不同的使用习惯
- 避免过长等待：最大 10 秒限制保证用户不会等太久

### 决策 4：冲突解决策略

**选择**：用户选择模式（弹出对话框）

**选项**：
- 使用本地版本
- 使用远端版本
- 合并编辑
- 稍后处理

**理由**：
- 尊重用户数据：避免自动覆盖导致数据丢失
- 透明决策：用户清楚选择了什么

### 决策 5：离线恢复策略

**选择**：持久化待同步列表 + 网络恢复自动重试

**持久化文件**：`~/.vnote/sync_pending.json`

```json
{
  "pendingFiles": [
    {
      "path": "notes/example.md",
      "content": "...",
      "sha": "abc123",
      "timestamp": 1700000000,
      "retryCount": 0
    }
  ],
  "lastSyncTime": 1700000000
}
```

**理由**：
- 可靠性：应用崩溃或关机后不会丢失待同步数据
- 自动恢复：网络恢复后自动重试，用户无感知

### 决策 6：SHA 缓存管理

**选择**：在 pullFile 后立即更新 SHA

```cpp
void GiteeSyncService::pullFile(...) {
    // ... existing pull logic ...
    m_fileShaMap[path] = response["sha"].toString(); // 新增
}
```

**理由**：
- 修复现有 bug：当前 pullFile 后 SHA 未更新
- 简单直接：无需额外的缓存同步机制

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| R1 | 批量窗口内文件内容可能被后续修改覆盖 | 使用 "last-write-wins" 策略，同步时取最新内容 |
| R2 | 离线文件列表可能导致磁盘占用增长 | 定期清理超过 7 天的待同步记录 |
| R3 | 状态栏指示器可能影响性能 | 使用 Qt 信号槽异步更新，避免阻塞 UI |
| R4 | 配置界面增加代码复杂度 | 使用现有 ConfigMgr 框架，保持风格一致 |

## Migration Plan

1. **Phase 1：核心调度器**
   - 新增 `SmartSyncScheduler` 类
   - 实现事件路由和防抖机制
   - 保持 `GiteeSyncService` 接口兼容

2. **Phase 2：状态管理**
   - 新增 `SyncStatusManager` 类
   - 实现文件状态机
   - 添加状态栏指示器

3. **Phase 3：持久化与恢复**
   - 实现 `sync_pending.json` 读写
   - 添加网络状态监听
   - 实现自动重试逻辑

4. **Phase 4：UI 增强**
   - 新增同步管理中心视图
   - 添加配置界面
   - 完善错误提示

## Open Questions

1. **Q1**：批量窗口如何与用户的"立即同步"操作交互？是否取消当前防抖队列？
   - 建议：Ctrl+S 时立即执行当前队列，并清空防抖计时器

2. **Q2**：同步失败达到最大重试次数后的处理策略？
   - 建议：移入"失败列表"，等待用户手动处理或下次网络恢复

3. **Q3**：是否需要显示同步进度（如 3/10 文件已同步）？
   - 建议：Phase 2 实现基本进度显示，Phase 3 完善细节
