## Context

当前 VNote 使用本地文件系统存储笔记数据，通过 git 命令实现版本控制和多设备同步。这种方式要求用户手动执行 git add/commit/push 操作，容易遗忘且操作繁琐。项目已有 GiteeImageHost 组件实现了 Gitee API 的认证和调用，提供了可复用的基础设施。VNote 的架构采用 Buffer → VXNodeFile → LocalNotebookBackend → 本地文件系统的分层结构，vx.json 作为节点元数据文件记录子节点列表、标签、时间等信息。

## Goals / Non-Goals

**Goals:**
- 通过 Gitee API 自动同步笔记数据，无需手动 git 操作
- 实现多设备间无缝协作，用户在公司编辑后回家可继续查看最新内容
- 提供灵活的拉取策略（打开笔记本时、展开目录时、打开文件时）
- 使用合并窗口机制避免频繁 API 调用，减少 Gitee API 速率限制影响
- 保留 vx.json 同步到 Gitee，确保多设备间节点元数据一致

**Non-Goals:**
- 不处理冲突解决场景（暂不考虑多设备同时编辑同一文件的冲突）
- 不修改现有 git 工作流（git 命令仍可正常使用）
- 不支持其他 Git 托管平台（仅支持 Gitee）
- 不修改 vx.json 的格式和用途（保持向后兼容）

## Decisions

### 1. 架构设计：GiteeSyncService 单例组件
采用单例模式的 GiteeSyncService 协调所有同步操作，包含三个子模块：
- **GiteeApi**: 封装 Gitee API v5 Contents 接口（创建/更新/获取/删除文件）
- **ShaCache**: 缓存远端文件 SHA，避免重复获取
- **PushQueue**: 管理推送队列，实现合并窗口机制

**为什么选择单例模式：**
- 与 VNote 现有架构一致（ThemeMgr、TaskMgr、NotebookMgr 均为单例）
- 全局共享 SHA 缓存和推送队列，避免重复初始化
- 便于在 VXNodeFile、LocalNotebookBackend、ConfigMgr 等多处访问

**替代方案考虑：**
- 多实例模式：每个 Notebook 一个实例 → 增加复杂度，SHA 缓存无法共享
- 依赖注入模式：通过构造函数传入 → 改动较大，与现有架构不一致

### 2. 拉取策略设计
三个接入点分别触发不同层级的拉取：
1. **VXNodeFile::read() 前**：打开文件时拉取该文件内容
2. **LocalNotebookBackend::loadRootNode() 前**：打开笔记本时拉取根目录
3. **ConfigMgr::loadNode() 前**：加载节点配置时拉取该节点的 vx.json

**为什么这样设计：**
- 按需拉取，避免一次性下载整个仓库
- 与用户操作流程自然契合（打开→拉取，展开→拉取）
- 最小化网络请求，提升响应速度

**替代方案考虑：**
- 定时全量拉取：消耗大，不适用于大型仓库
- 后台增量拉取：实现复杂，可能产生冲突

### 3. 推送策略设计：合并窗口机制
用户保存文件后，将推送任务加入队列，启动倒计时定时器（默认 5 秒，可配置）。如果倒计时期间有新的保存操作，重置倒计时。倒计时结束后执行推送，队列中同一文件的多次修改只推送最后一次。

**队列策略：** PushQueue 无上限，保证数据完整性。应用关闭前检查队列状态，如有未推送任务则弹窗提示用户选择"等待同步"或"退出"。

**为什么选择合并窗口：**
- 避免频繁 API 调用，减少速率限制风险
- 提升用户体验，编辑过程中不频繁等待网络请求
- 与 Buffer 的 1 秒自动保存机制配合良好

**为什么队列无上限：**
- 保证数据完整性，不因队列超限丢弃用户修改
- 单条任务只存储文件路径（内容从磁盘读取），内存占用可控
- 避免频繁弹窗干扰用户编辑体验

**替代方案考虑：**
- 立即推送：频繁 API 调用，可能触发速率限制
- 定时批量推送：实时性差，用户关闭应用前可能丢失数据
- 有上限队列：超限丢弃修改，违背数据完整性目标

### 4. SHA 缓存机制
使用内存 Map 缓存文件路径 → SHA 映射，缓存策略：
- 首次推送文件时，调用 Gitee API 获取 SHA 并缓存
- 后续推送直接使用缓存 SHA
- 拉取文件/目录成功后，将返回的 SHA 更新到缓存
- 获取目录内容时，批量预缓存所有子文件/子目录的 SHA
- SHA 冲突（409）时自动获取最新 SHA 并重试一次推送
- 缓存有效期：应用生命周期内（关闭应用后清空）

**SHA 失效自动刷新策略：**
- 推送遇到 409 冲突时，重新调用 API 获取最新 SHA
- 用新 SHA 重试一次推送，最多重试 1 次
- 仍失败则记录日志并弹窗提示用户

**批量预缓存优化：**
- 获取目录内容 API 返回每个文件/目录的 SHA
- 一次性批量缓存所有 SHA，避免逐个获取
- 大幅减少后续文件修改时的网络请求

**为什么需要 SHA 缓存：**
- Gitee API 更新文件必须提供远端当前 SHA
- 避免每次推送前都调用 API 获取 SHA，减少网络请求
- 批量预缓存提升性能，减少应用重启后的首次推送延迟

**替代方案考虑：**
- 无缓存：每次推送前获取 SHA → 网络请求翻倍
- 持久化缓存：增加复杂度，且应用关闭后数据可能已过期

### 5. 配置管理
在 ConfigMgr 中新增 Gitee 同步配置项：
- `gitee_token`: Gitee 个人访问令牌
- `gitee_owner`: 仓库所有者（用户名/组织名）
- `gitee_repo`: 仓库名称
- `gitee_branch`: 分支名称（默认 "master"）
- `gitee_sync_enabled`: 是否启用同步（默认 false）
- `gitee_push_delay_seconds`: 推送倒计时秒数（默认 5）

**UI 配置界面：**
- 在设置界面新增 "Gitee 同步" 配置页面
- 提供文本框输入 token、owner、repo、branch
- 提供开关控制同步启用/禁用
- 提供数字输入框调整推送倒计时秒数（范围 1-30 秒）
- 配置变更时立即生效，无需重启应用

**为什么在 ConfigMgr 中配置：**
- 与 VNote 现有配置管理方式一致
- 用户可在设置界面统一管理
- 配置持久化，无需重复输入

### 6. 错误处理策略
- 网络错误：记录日志，弹窗提示用户，不阻塞主流程（本地保存成功即可），任务留在队列中等待重试
- API 错误（401/403）：记录日志，弹窗提示用户检查 token 配置，不重试该任务
- 速率限制（429）：队列任务等待重试（指数退避），记录日志并弹窗提示
- SHA 冲突（409）：自动获取最新 SHA 并重试一次，仍失败则记录日志并弹窗提示
- 应用关闭时检查队列：如有未推送任务，弹窗提示用户"有 N 个文件未同步，是否等待同步完成？"
- 其他 API 错误：记录日志，弹窗提示用户

**弹窗提示策略：**
- 同步成功：不在状态栏显示，静默成功
- 同步失败：弹窗提示错误详情（仅首次出现时，避免重复弹窗）
- 速率限制：弹窗提示并显示重试倒计时
- 配置错误：弹窗提示用户修正配置

**为什么选择非阻塞策略：**
- 保证本地操作不受网络问题影响
- 用户可继续编辑，后台异步重试
- 降低用户焦虑，提升体验

### 7. 参考 GiteeImageHost 实现
复用 GiteeImageHost 的以下内容：
- Gitee API 认证机制（Authorization header）
- 网络请求工具（NetworkUtils）
- 错误处理逻辑

**为什么不直接扩展 GiteeImageHost：**
- 职责不同：GiteeImageHost 专注图片托管，GiteeSyncService 专注笔记同步
- 独立演化：两个组件未来需求可能差异较大

## Risks / Trade-offs

### 1. 速率限制风险
Gitee API 有速率限制，高频推送可能触发限制。

**缓解措施：**
- 使用合并窗口机制减少请求频率
- 队列任务遇到速率限制时指数退避重试
- 记录日志，提示用户减少编辑频率

### 2. 网络依赖风险
无网络时无法同步，可能导致多设备数据不一致。

**缓解措施：**
- 本地操作不阻塞（非阻塞策略）
- 网络恢复后自动重试队列中的任务
- 应用关闭前检查队列状态，提示用户等待同步完成

### 3. vx.json 同步的仓库整洁问题
vx.json 包含内部元数据，推送到 Gitee 后仓库不干净。

**缓解措施：**
- 用户已选择路径 A（同步 vx.json），接受此 trade-off
- 未来可考虑在 .gitignore 中忽略 vx.json（需配合路径 B 方案）

### 4. 冲突处理未实现
多设备同时编辑同一文件时可能产生冲突。

**缓解措施：**
- 暂不处理冲突场景（用户需求明确）
- 记录日志，提示用户手动检查
- 未来可引入冲突检测和合并机制

### 5. 增加代码复杂度
新增多个模块和接入点，增加维护成本。

**缓解措施：**
- 单元测试覆盖核心逻辑
- 清晰的模块边界和接口定义
- 详细注释和文档

## Migration Plan

### 部署步骤
1. 新增代码模块（GiteeSyncService、ShaCache、PushQueue、GiteeApi）
2. 修改现有代码接入同步逻辑（VXNodeFile、LocalNotebookBackend、ConfigMgr）
3. 在 ConfigMgr 中添加配置项
4. 编译测试，确保功能正常

### 回滚策略
- 配置项默认关闭（`gitee_sync_enabled: false`），不影响现有用户
- 代码改动可独立回滚（新模块删除，修改处还原）
- 本地文件系统不受影响，git 工作流仍可用

### 兼容性
- 向后兼容：不影响现有 git 工作流
- 向前兼容：配置项可独立管理，未来支持其他 Git 平台
